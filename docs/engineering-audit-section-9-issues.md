## 1. What is broken or not working correctly right now?

These are things that exist in the app, appear to work, but will fail when used.

---

### Endorsement edits fail silently
The edit page (`/endorse/edit/{id}?token={token}`) exists, is linked from the submission confirmation screen, and looks functional. But the Firestore security rules include `allow update, delete: if false` on the `endorsements` collection. Every time someone clicks Save on that page, Firestore silently rejects the write. The user gets no useful error — the page just doesn't update. Anyone who submitted an endorsement and later tries to change it cannot do so.

---

### The "46 delegates" milestone never fires
The volunteer dashboard shows a milestone banner when committed+locked delegate count crosses key thresholds. The message for 46 reads: *"WE HAVE THE MINIMUM TO WIN."* But the Cloud Function that fires milestones checks the list `[10, 20, 30, 40, 50, 53, 60, 70]`. The number 46 is not in it. That milestone will never trigger no matter what the actual count is.

---

### Call script wizard saves are likely being rejected by Firestore
When a volunteer completes the six-step guided call flow, the app writes to the `callScriptLogs` collection. That collection has no entry in the Firestore security rules file. Firestore denies all access to any collection not explicitly covered by a rule. This means wizard completions are likely failing silently — the volunteer sees the success screen, but nothing is saved.

---

### The scheduled Cloud Functions are likely failing every day
Two Cloud Functions run on a schedule:
- `staleDelegateAlert` — runs every 24 hours, finds delegates not contacted in 14+ days
- `staleDelegateSurveyAlert` — runs daily at 8 AM MT, flags incomplete surveys

Both execute Firestore queries that require composite indexes (multiple `where` clauses on different fields, or queries on nested fields like `survey.startedAt`). No `firestore.indexes.json` file exists in the repo. If those indexes were not manually created in the Firebase console, both functions error out on every scheduled run. The stale delegate list and the survey follow-up flags they write to the dashboard would never update. There would be no visible indication anything was wrong — the dashboard would just show stale or missing data.

---

### `campaignStats/live` may not exist
The Cloud Function `onContactLogCreated` calls `update()` on `campaignStats/live` after every contact log. If that document does not exist in Firestore, `update()` throws an error and the entire function fails — meaning no delegate stage history gets written, no leaning counts get recalculated, and no milestones ever fire. The document has no initialization code anywhere; it must have been created manually at some point. If it was ever accidentally deleted, the whole contact log pipeline would silently break.

---

### Convention dates are stale throughout the app
The convention was April 11, 2026. That date is hardcoded in at least six places:
- The scoreboard's "days until convention" counter (now showing a negative number)
- The convention thank-you email template and SMS
- The post-convention nominee email
- The AI message polish prompt (`"The convention is April 11"` at the end of every generated message)
- The "Calling All Delegates" email template
- The call script wizard closing script

Every outreach message generated by the AI currently tells a delegate that the convention is April 11 and asks them to come vote. The convention has passed.

---

### The `/admin` route shows the admin UI to any logged-in user
Any volunteer who navigates to `wileyfor21.com/admin` will see the full admin dashboard load — the delegate table with all contact information, the CSV importer, the convention email tools, everything. Firestore rules will reject any destructive write they attempt, but all delegate names, phone numbers, email addresses, precincts, and contact history are visible to anyone with a volunteer account. The route needs a role check before it renders.

## 2. What was built quickly and needs to be done properly?

These things work well enough to have gotten through the convention, but they carry real risk or will cause pain as the campaign continues.

---

### CSV import has no safety net
The delegate import flow is the most operationally risky part of the system. It has no duplicate detection — importing the same file twice doubles every delegate record. It has no row-level validation — a phone number in the email column, a blank required field, a differently-formatted name will all be accepted silently. And if any row throws an error mid-import, the entire batch stops with no rollback — some delegates are in, some are not, and there is no way to know which without manually counting.

The prescribed workaround — "Clear All Delegates before reimporting" — is itself a single-click operation that permanently deletes the entire delegate collection with no undo. That is a dangerous combination.

---

### There are two contact log forms and they behave differently
The pop-up `ContactLogModal` and the inline log form inside `DelegateCard` are separate implementations. The modal has an explicit "advance stage?" question that the volunteer must answer. The inline form does not — it auto-advances based on candidate rankings instead. A volunteer using one will get different stage-transition behavior than one using the other, with no explanation of the difference. The two should be a single implementation.

---

### The call script wizard and contact log are disconnected
After completing the six-step call wizard, the app opens the contact log form for the volunteer to also fill out. The notes from the wizard are not passed to that form. The volunteer types their outcome and details again from scratch. The wizard saves to `callScriptLogs`; the contact log saves to `contactLogs`. They are two separate records with no link between them. The Cloud Function only triggers on `contactLogs`, so a call that used the wizard but skipped the follow-up contact log form does not increment stats, does not update leaning, and does not fire milestones.

---

### Volunteer outreach story is stored in the browser only
When a volunteer fills out their personal "why I support Aaron" story and generates an AI-polished message, that data is saved to `localStorage` under the key `wileyfor21_volunteer_story`. It is not saved to Firestore. If they clear their cache, switch devices, or use a different browser, their story is gone and they have to write it again. At scale across many volunteers, this creates inconsistent experience and lost personalization work.

---

### Next action dates are collected but never used
Every contact log asks the volunteer to select a next action — "Follow up in 3 days," "Follow up in 1 week," and so on. For those two options, a `nextContactDate` is calculated and written to the contact log document. That date is never displayed anywhere in the volunteer dashboard, never surfaced as a filter or sort, and never used to trigger a reminder. The system collects follow-up commitments from volunteers and then does nothing with them.

---

### The `hub/` subdirectory is dead code living alongside the active app
There are two separate React apps in this repo: the current app in `src/` and a legacy version in `hub/src/`. They have different dependency versions, different Firebase SDK versions (12 vs 10), different Tailwind configurations, and partially overlapping functionality. The `hub/` app is not deployed anywhere, but it still exists in the repo, has its own `node_modules`, and can be confused with the active codebase. The Firebase Cloud Functions that power the live app live inside `hub/functions/` — so the folder cannot simply be deleted — but the `hub/src/` React app is dead and should be removed.

---

### The `assignedDelegates` field on volunteer documents is unused
When a volunteer signs up, their document is created with `assignedDelegates: []`. This field is never written to or read from by any part of the app. The actual assignment system works in the opposite direction — volunteer UIDs are stored in `delegates.assignedTo[]`. The `assignedDelegates` field is noise. It is also mildly confusing because it implies a different data model than what actually exists.

## 3. What parts of the codebase concern me most?

These are not bugs but structural patterns that will create problems as the campaign grows.

---

### There is no error monitoring
No Sentry, no Datadog, no Firebase Crashlytics, no alerting of any kind. If a Cloud Function starts failing — because an index is missing, because `campaignStats/live` does not exist, because Firestore rules reject a write — no one will know unless they actively look at the Firebase Functions logs. Given that the scheduled functions run daily and their failures would silently break the stale delegate and survey follow-up features, this is a real operational gap. Right now the only way to know if something is broken is to notice that data stopped updating.

---

### No indexes are tracked in code
The `firestore.indexes.json` file that would define all required Firestore composite indexes does not exist. Any indexes that are in production were created manually in the Firebase console and are not reflected in the repo. This means:
- There is no way to know from the codebase which indexes exist
- If the Firebase project were recreated or the indexes accidentally deleted, several queries would start silently failing
- New developers (or Claude) cannot know from the code alone whether queries will work in production

---

### The admin dashboard is a 2,500-line file
`AdminDashboard.jsx` is the single largest file in the codebase at approximately 2,500 lines. It contains the delegate table, CSV importer, stage editor, volunteer assignment, all three email campaign sections, the contact distribution tool, and the scoreboard — as one monolithic component. Adding a new admin feature, fixing a bug in one section, or understanding what the file does requires navigating thousands of lines of mixed logic and markup. This is the file most likely to introduce a regression when anything touches it.

---

### The Netlify form and Firebase volunteer database are two separate silos with no bridge
People who sign up on the public landing page go into Netlify. People who create accounts in the hub go into Firebase. These are completely separate systems with no connection. Someone can sign up on the website, receive no follow-up, and never appear in the hub. Or a volunteer can be in the hub but have signed up through a channel that bypassed the website form. There is no single source of truth for "everyone who has expressed interest in volunteering."

## 4. Security concerns to address

---

### Any volunteer can see all delegate contact information
Addressed above under the `/admin` route issue, but worth restating plainly: right now, a volunteer who knows the URL can navigate to the admin dashboard and see every delegate's name, phone number, email, home address, precinct, leaning, contact history, and notes. Firestore rules prevent them from writing anything, but there is no read restriction at the route level. This should be fixed before any future work adds more sensitive data to delegate records.

---

### The unauthenticated delegate survey link uses a Firestore document ID as the access token
When a volunteer sends a delegate the direct survey link (`/delegate/survey?delegate={id}`), the `delegate` parameter is the Firestore document ID. Firestore document IDs are not secrets — they are short alphanumeric strings that are not generated with cryptographic randomness. Anyone who knows or guesses a delegate's ID can open that link and overwrite their survey responses. There is no expiry, no revocation, and no audit trail of who opened the link. A proper implementation would generate a dedicated, cryptographically random token stored separately from the document ID.

---

### No email verification on signup
When a volunteer or delegate creates an account, Firebase Auth does not require them to verify their email address. Someone can sign up with `anyone_elses_email@example.com` and get immediate access to the hub. This is a minor risk at campaign scale but is worth noting — email verification is a one-line Firebase Auth configuration change.

---

### Endorsement submissions go directly to the public wall with no review
Anyone can submit any name, title, and text to the endorsement form and it appears immediately on Aaron's public campaign website. There is no spam filter, no moderation queue, and no admin review step. Removing a bad submission requires direct Firestore console access. At the scale of a district campaign this is a manageable risk, but it is worth having a plan for it.

---

### The Anthropic API key is handled correctly — nothing to fix there
The `ANTHROPIC_API_KEY` is stored in Netlify environment variables and is only accessed server-side inside the Netlify Function. It is never sent to the browser. This is the right pattern and does not need to change.
